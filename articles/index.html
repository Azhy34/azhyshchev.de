Common thread across all these projects

The real engineering is not in the LLM call. It is in what surrounds it: controlled ingestion, deterministic fallbacks, idempotent pipelines, and structured storage. That combination is what separates production automation from demos.

Case Study · RAG · AI Agents · Vibe Coding

Hybrid RAG Telegram Sales Agent

Challenge: Cost vs. Latency

A static 57 MB PDF product catalog — 55K+ tokens. Using Google Gemini File Search for its native multimodal capabilities, the raw catalog hit the "Hot Cache" pricing tier (~$45/mo) and caused high latency. Throwing money at the problem was not the solution.

Optimization

Model pivot: Benchmarked models and switched to gemini-2.5-flash. Result: sub-second response times and minimal API costs — completely eliminating the need for expensive server-side caching.

Hybrid RAG Architecture — the technical core

To prevent hallucinations and ensure accuracy, data retrieval is split into two layers:

Supabase (PostgreSQL + pgvector): Stores the "Hard Data" — SKU, exact pricing, stock status, image URLs. Used for precise card generation. The bot never invents a price.

Gemini File Search: Handles the "Soft Context". Queries like "What is good for joint pain?" or "How do I apply this?" retrieve semantic answers from the unstructured PDF text.

AI Asset Pipeline

Low-res PDF crops were unusable for a sales experience. Built an automated pipeline where Gemini 3 Pro Vision upscales and regenerates product images into studio-quality assets on the fly.

Case Study · Automation · LLM Pipeline

Invoice Automation Pipeline

The problem

Invoice processing from email quietly eats hours every month. Incoming invoices arrive as PDFs from multiple suppliers — Miele, Siemens, Liebherr, Bosch, Smeg — in different formats, different file names. Someone has to open the inbox, find attachments, classify the document, extract data, file it in the right folder, and enter it into accounting. Every week.

Architecture

Two-layer LLM design: Gemini is not the only source of truth. A deterministic baseline parser (regex, keyword triggers, field extraction) runs first. Gemini 2.5 Flash verifies and normalises on top. When the API is down — the fallback works.

Two-level deduplication: Attachment-level (message ID + hash) and business-level (supplier + invoice number). Handles re-forwarded emails, cron restarts, and duplicate PDFs in one message.

Status model: Every attachment gets a processing_log record regardless of outcome. Nothing is silently lost.

Key metrics
0 Manual hours / week
100% Automated filing
$0 Infrastructure cost (GitHub Actions)

Technical Insight · RAG Architecture

From Data Mirroring to Knowledge Synthesis

The problem

How do you minimize LLM hallucinations when working with complex technical catalogs? The answer isn't in better prompting — it's in the shift from Data Mirroring to Knowledge Synthesis.

My first mistake was vectorizing raw PDF catalogs directly. I expected similarity search to handle dense, multi-column technical datasheets. The result: a "calculator without a brain" — the bot found numbers but lacked expert intuition. It wasted tokens parsing raw data without providing real advice.

The architectural shift: from data to meaning

1. Knowledge Synthesis (LCEL-driven)
Instead of basic chunking, LCEL chains distill raw PDF specs into expert insights during ETL. The database stores processed meanings, not unstructured text.

2. Two-Layer Storage (Supabase + LCEL)
Child chunks (400 tokens) — semantic hooks (FAQs, usage scenarios, USPs) in pgvector. Instead of "DualSteam" → "Ideal for healthy cooking: preserves all vitamins and color." Instead of "TwinDos" → "Effortless laundry: auto-dispenses perfect detergent amount."
Parent documents — full JSON records in PostgreSQL. RunnableParallel fetches SQL facts while checking the cache, ensuring sub-second response times.

3. Quality Control
Dry-run every category on a single item to calibrate the prompt — 100% accuracy before mass vectorization.

Impact

The LLM now gets: User Query + Expert Scenario + Precise SQL Data.

Zero Hallucinations
Clean Context windows
<1s Response time

Case Study · RAG · AI Agent · E-commerce

Web Sales Assistant — da-vinchi.pl

The problem

A Polish wallpaper e-commerce store needed a sales assistant that could handle real buyer queries — style recommendations, price calculations, FAQ, direct SKU lookups — without hallucinating prices or inventing product links. A plain LLM over documents wouldn't work: too imprecise on facts, too blind to semantics.

What the agent handles — without the sales team

From semantic product discovery to policy lookups, the agent covers the full pre-sales conversation — 24/7, with no manual intervention.

"Show me wallpapers for a kids room with flowers"
Semantic search across 1,200 SKUs — instant shortlist with prices, no manual filtering
"What's the price for SKU 000389, does it come in a set?"
Exact catalog lookup — precise specs, no estimating, no guessing, no errors
"How do I hang it? What glue do I need?"
Knowledge base answers from company materials — installation guides served instantly
"What's your return policy?"
FAQ retrieval with grounded company info — no fabricated policies, only verified answers
Hybrid RAG architecture

The core architectural decision: split knowledge into two truth layers.

SQL (Supabase PostgreSQL): Hard facts — SKU, final_price, stock_status, route, colors, rooms. The agent never constructs a URL or price from memory. Everything comes from structured data.

pgvector (semantic KB): Soft knowledge — style guides, room recommendations, FAQ, installation, sales process. Queries like "something calm for a bedroom" are understood semantically first, then translated into SQL filters.

LangGraph agent — 5 tools

search_knowledge — semantic retrieval from pgvector (style mapping, room guides, FAQ)

search_catalog — faceted SQL search with fallback logic (relaxes filters progressively if results are too few)

get_product_details — direct lookup by SKU, product_id, or URL slug — the main anti-hallucination tool

compare_styles — compares two styles for a user's specific need via retrieval

calculate_wall_price — computes total cost by wall dimensions including waste margin

ETL + knowledge base

Two independent pipelines: catalog import from CSV (355 active SKUs after dedup and out-of-stock filtering) and knowledge base build from structured markdown (142 chunks across 5 semantic types: faq, technical, style_mapping, room_guide, sales).

Chunking strategy varies by document type — FAQ splits per question, room guides carry structured metadata, style mappings include color and category fields.

Case Study · Full-Stack · AI Music · SaaS

AI Music SaaS — hitdlaciebie.pl

Full-stack AI music SaaS built for a client in two weeks with zero prior experience in AI music generation. hitdlaciebie.pl generates personalized songs in Polish: user describes the occasion and who the song is for, AI writes lyrics and sings a full track, the MP3 lands in the personal cabinet.

2 weeks Idea to production
4 APIs Tested before shipping
2 LLMs In sequence for lyrics

Lyrics pipeline — why two models instead of one

Generation uses a two-model chain via OpenRouter. Gemini 2.5 Flash (temperature 0.8) generates a raw creative draft — imaginative but structurally inconsistent. Claude Sonnet 4.6 (temperature 0.35) polishes the draft — enforces rhyme scheme, formats [Verse][Chorus] tags that Suno needs.

Single-model attempts failed in both directions: high temperature broke structure, low temperature killed creativity.

Music API selection — tested four, shipped one

fal.ai / Minimax — fast queue, but Polish phonetics were weak. Vocals sounded synthesized on vowel-heavy words.

ElevenLabs Music — good timbre, but no verse/chorus structure awareness. Treated the entire lyric as a single block.

Udio — promising quality, unstable API during testing. Requests failed silently.

Suno V5.5 — natural Polish vocals, correct structure from lyrics tags, consistent generation. Shipped to production. All four remain as registered providers — architecture is provider-agnostic.

What broke along the way

Problem 1 — two days lost to a UI checkbox. Token credits were not appearing after test purchases. Reviewed the entire webhook flow: HMAC verification, DB writes, token logic — everything was correct. The actual issue: Shopify test mode requires manually clicking "Mark as paid" in admin. The code worked from day one.

Problem 2 — SSRF in the audio proxy. Suno CDN URLs expire, so I built a proxy-audio route. Missed that it accepted arbitrary URLs. Fixed with strict URL.hostname validation — only *.suno.ai over HTTPS. Legacy fal.ai URLs in DB broke after the fix — extended allowlist for backward compatibility.

Problem 3 — duplicate webhook delivery. Shopify can fire the same event twice. Fixed with unique constraint on shopify_order_id + ON CONFLICT DO NOTHING. Idempotent by design.
